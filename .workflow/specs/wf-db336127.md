# Spec: wf-db336127 — Add Emoji Reactions to Messages

## User Story
**As a** developer
**I want** emoji reaction support on messages
**So that** users can react to messages with emoji (like, heart, thumbs up, etc.)

## Design Decision
Separate `Reaction` Prisma model with composite unique constraint on `[messageId, userId, emoji]` — one reaction type per user per message.

## Acceptance Criteria

### Scenario 1: Add a reaction
**Given** POST /api/reaction
**When** messageId, userId, and emoji are provided
**Then** a Reaction record is created
**And** returns `{ data: reaction }`
**And** throws NotFoundException if message not found or deleted
**And** throws ConflictException if user already reacted with same emoji on same message
**And** broadcasts `reactionAdded` event via WebSocket to the message's conversation

### Scenario 2: Remove a reaction
**Given** DELETE /api/reaction/:id
**When** a valid reaction ID is provided
**Then** the Reaction record is hard-deleted
**And** returns `{ data: reaction, message: 'Reaction removed' }`
**And** throws NotFoundException if reaction not found
**And** broadcasts `reactionRemoved` event via WebSocket to the message's conversation

### Scenario 3: List reactions for a message
**Given** GET /api/reaction?messageId=<uuid>
**When** a valid messageId query param is provided
**Then** returns all reactions for that message
**And** returns `{ data: reactions[] }`

### Scenario 4: WebSocket broadcast
**Given** a reaction is added or removed
**When** the REST endpoint completes
**Then** broadcastToRoom sends event to the message's conversationId
**And** event payload: `{ messageId, userId, emoji, type: 'added'|'removed' }`

### Scenario 5: Unit tests
**Given** the reaction module is created
**When** tests are run
**Then** all controller + service methods are covered
**And** conflict/not-found edge cases are tested

## Implementation Steps

### Step 1: Prisma schema
- Create `prisma/models/reaction.prisma` with: id, messageId, userId, emoji, createdAt
- Add `@@unique([messageId, userId, emoji])` composite constraint
- Add `reactions Reaction[]` relation on Message model
- Run prisma:merge, prisma:generate, create migration

### Step 2: Reaction module scaffold
- Create `src/reaction/reaction.module.ts`
- Create `src/reaction/reaction.service.ts` — thin Prisma wrapper with create(), findAll(), findOne(), remove()
- Register in AppModule

### Step 3: DTOs
- Create `src/reaction/dto/create-reaction.dto.ts` — messageId, userId, emoji (all required)
- Swagger decorators + class-validator

### Step 4: Controllers
- Create `src/reaction/controllers/base.controller.ts` — injects ReactionService, ChatGateway, MessageService
- Create `src/reaction/controllers/create-reaction.controller.ts` — POST /api/reaction
  - Validate message exists and not deleted
  - Handle unique constraint violation → ConflictException
  - Broadcast reactionAdded event
- Create `src/reaction/controllers/delete-reaction.controller.ts` — DELETE /api/reaction/:id
  - Validate reaction exists
  - Hard delete (not soft delete)
  - Broadcast reactionRemoved event
- Create `src/reaction/controllers/list-reaction.controller.ts` — GET /api/reaction?messageId=<uuid>

### Step 5: Module wiring
- Import ChatGatewayModule in ReactionModule (for WebSocket broadcast)
- Import MessageModule in ReactionModule (for message validation)
- Import ReactionModule in AppModule

### Step 6: Tests
- Create `src/reaction/reaction.spec.ts` — unit tests for all controllers and service
