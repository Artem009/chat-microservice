# Spec: wf-728525a3 — Update E2E Tests for New Endpoints and WebSocket Events

## User Story
**As a** developer
**I want** E2E tests covering all endpoints and WebSocket events added after wf-0bfcc892
**So that** the full API surface has integration-level test coverage

## Description
E2E tests were written during wf-0bfcc892 when 16 REST endpoints and 5 WS events existed. Since then, 4 new REST endpoints (Reaction 3, Mention 1, Thread 1) and 6 new WS broadcast events were added without E2E coverage. This task brings E2E coverage from 64% to 100%.

## Acceptance Criteria

### Scenario 1: Reaction E2E tests (3 endpoints)
**Given** the test app is running with mock Prisma
**When** POST /api/reaction is called with valid messageId, userId, emoji
**Then** returns 201 with `{ data: reaction }`
**And** returns 400 when required fields are missing
**And** returns 409 when duplicate reaction (P2002 Prisma error)

**When** GET /api/reaction?messageId=xxx is called
**Then** returns 200 with `{ data: reactions[] }`

**When** DELETE /api/reaction/:id is called
**Then** returns 200 with `{ data: reaction, message: 'Reaction deleted' }`
**And** returns 404 when reaction not found

### Scenario 2: Mention E2E test (1 endpoint)
**Given** the test app is running with mock Prisma
**When** GET /api/mention?messageId=xxx is called
**Then** returns 200 with `{ data: mentions[] }`
**When** GET /api/mention?userId=xxx is called
**Then** returns 200 with `{ data: mentions[] }`
**When** GET /api/mention is called without params
**Then** returns 200 with `{ data: [] }`

### Scenario 3: Thread E2E test (1 endpoint)
**Given** the test app is running with mock Prisma
**When** GET /api/message/thread/:parentMessageId is called
**Then** returns 200 with `{ data: replies[] }`
**When** no replies exist
**Then** returns 200 with `{ data: [] }`

### Scenario 4: WebSocket broadcast E2E tests (6 events)
**Given** two WS clients connected and joined to the same conversation
**When** a newMessage event is triggered via REST POST /api/message
**Then** clients in the room receive the `newMessage` broadcast

**When** a threadReply event is triggered via REST POST /api/message with parentMessageId
**Then** clients receive the `threadReply` broadcast with `{ parentMessageId, reply }`

**When** a readReceipt event is triggered via REST POST /api/message/read
**Then** clients receive the `readReceipt` broadcast

**When** a reactionAdded event is triggered via REST POST /api/reaction
**Then** clients receive the `reactionAdded` broadcast

**When** a reactionRemoved event is triggered via REST DELETE /api/reaction/:id
**Then** clients receive the `reactionRemoved` broadcast

**When** a userMentioned event is triggered via REST POST /api/message with @uuid
**Then** clients receive the `userMentioned` broadcast

### Scenario 5: Update e2e-setup mock to include new Prisma models
**Given** the test helper `test/helpers/e2e-setup.ts`
**When** new test files need reaction, mention models
**Then** `createMockPrismaService()` includes `reaction` and `mention` mock objects

## Implementation Steps

### Step 1: Update e2e-setup.ts
- Add `reaction` model mock (create, findMany, findUnique, update, delete)
- Add `mention` model mock (createManyAndReturn, findMany)

### Step 2: Create reaction.e2e-spec.ts
- POST /api/reaction (201 success, 400 validation, 409 conflict)
- GET /api/reaction?messageId= (200 list)
- DELETE /api/reaction/:id (200 success, 404 not found)

### Step 3: Create mention.e2e-spec.ts
- GET /api/mention?messageId= (200)
- GET /api/mention?userId= (200)
- GET /api/mention without params (200 empty)

### Step 4: Add thread tests to message.e2e-spec.ts
- GET /api/message/thread/:parentMessageId (200 with replies, 200 empty)

### Step 5: Add WS broadcast tests to websocket.e2e-spec.ts
- newMessage broadcast (via REST→WS bridge)
- threadReply broadcast
- readReceipt broadcast
- reactionAdded broadcast
- reactionRemoved broadcast
- userMentioned broadcast
